Continuous Integration

El proceso de trabajo de Orient Software es rápido y eficiente para producir código para las necesidades de su negocio. Para garantizar que sus necesidades se cumplan de la mejor manera posible, nuestro proceso se centra en mantener y mejorar la calidad del software que desarrollamos y probamos. Pero, ¿qué significa un enfoque de calidad en nuestro trabajo diario? Echemos un vistazo a la práctica ágil de la integración continua.

Introducción a CI


La integración continua (IC) significa que cada cambio en el código se agrega al proyecto poco después de su escritura, y luego se prueba inmediatamente para detectar defectos. Esto garantiza que un cambio de código defectuoso se detecte inmediatamente y si algo "rompe" el producto, puede corregirse rápidamente, ya sea volviendo el cambio o buscando y corrigiendo el error en el nuevo código enviado.

CI se considera una de las prácticas de desarrollo de software Agile clave, que se origina en el llamado enfoque de Programación Extrema (XP). Se ha vuelto popular gracias a la presentación de beneficios medibles tanto para los proveedores de software como para los clientes con una mejor calidad, un tiempo de comercialización más rápido, menores costos y niveles de estrés reducidos durante todo el proceso de desarrollo del software.

Prerequisites

Aunque la integración continua puede parecer un cambio importante en la forma de ejecutar el desarrollo de software, el proceso en sí es bastante sencillo de seguir. Se basa en el Proceso de compilación automatizada, pero en lugar de simplemente ejecutar la compilación automatizada en la propia máquina local de un desarrollador, CI siempre involucra a todo el equipo o a todo el proyecto.

La Integración continua normalmente requiere 2 servidores: uno para almacenar el código fuente del proyecto, y el otro para ejecutar el software que realmente hace la integración de manera continua construyendo el proyecto automáticamente desde el código fuente más nuevo, y luego ejecutando las pruebas automatizadas. Dependiendo de la configuración del proyecto, estos servidores también pueden residir en la misma máquina física.

Version control

Cada desarrollador envía, o confirma, los cambios del código en el que han estado trabajando al repositorio en el servidor de administración de código fuente, para que todos los demás desarrolladores tengan acceso a los cambios más recientes. Esta es una práctica normal de control de versiones que es una necesidad para cualquier proyecto profesional de desarrollo de software.

En la integración continua, el servidor de CI supervisa el repositorio de código fuente para ver los cambios. En la práctica, esto puede significar que el repositorio notifica al servidor de CI cuando cambia y desencadena el proceso de CI, o que el servidor de CI accede periódicamente al repositorio y descubre si se ha modificado mientras tanto. El último se llama sondeo, y se puede hacer con cualquier intervalo, por ejemplo, una vez por minuto.

Integration

Una vez que el servidor de CI recibe información de que el código fuente ha sido cambiado, accede al repositorio, descarga los archivos modificados y los integra en la copia del código de proyecto del servidor de CI. Luego se ejecuta la compilación automática y se registra el resultado (éxito o falla). Después de una compilación exitosa, se ejecuta un conjunto de pruebas automatizadas del proyecto y nuevamente se registran los resultados, a menudo usando una escala de éxito de 3 niveles, inestable (algunos problemas detectados) o falla (pruebas fallidas u otros errores). Además, el servidor de CI también puede realizar otras tareas, como empaquetar el producto integrado para estar listo para la implementación o incluso implementar el software.

El estado final de la construcción y las pruebas se registra como el estado resultante, y si falla, los miembros del equipo son notificados para que el problema se pueda solucionar lo antes posible. Otros informes, como las mediciones de rendimiento, también se pueden publicar. Después de finalizar, el servidor está listo para comenzar la siguiente integración, una vez que haya nuevos cambios disponibles.

Tools

Existe una gran variedad de productos de software de integración continua disponibles. La elección de las herramientas depende de las habilidades y experiencia de la empresa y los desarrolladores, así como de los requisitos del cliente. Igualmente importante es el entorno de desarrollo y también el (los) entorno (s) objetivo (s): bajo qué sistema (s) operativo (es) funcionará el producto, cuáles son los lenguajes y marcos de programación utilizados, y qué software se integrará al producto.

En Orient Software, utilizamos principalmente dos herramientas de CI diferentes: Team Foundation Server (TFS) de Microsoft para proyectos .NET que también tienen su código fuente almacenado en TFS, y la herramienta de código abierto Jenkins CI para todo tipo de proyectos, cuyo código fuente es mayormente almacenado en repositorios Git. Podemos recomendar cualquiera de estas herramientas a nuestros clientes.

Advantages

Las ventajas de la integración continua son muchas, pero en última instancia están todas relacionadas con la calidad del software.

Cuando todo el nuevo código está integrado y pasa por pruebas, es muy probable que cualquier error de ruptura u otros defectos importantes se capturen rápidamente. De esta forma, tales errores no pueden permanecer ocultos por mucho tiempo. La detección rápida de errores permite aplicar correcciones casi de inmediato, antes de que los errores puedan afectar a otros módulos del sistema. Reparar errores de software al principio del desarrollo también es mucho menos costoso que solucionarlos días o incluso semanas después, una vez que el QC o el usuario final los haya encontrado en pruebas manuales o en uso regular. El ciclo de retroalimentación rápido ahorra recursos, reduce costos y conduce a beneficios financieros directos.

Otro aspecto clave de la calidad del software es la capacidad de mantenimiento. La práctica de integración continua alienta a los desarrolladores a dividir su trabajo en tareas más pequeñas (y más fáciles de entender) y se compromete a evitar romper la compilación y causar errores. Esto admite el proceso de revisión de código y hace que el código resultante sea más fácil de leer y modificar en el futuro. Es menos probable que un código más fácil de mantener contenga errores ocultos, y realizar cambios en él más tarde, por ejemplo, agregar funciones, será más rápido. CI por lo tanto promueve el ahorro también en el futuro.

Orient Software approach

Orient Software aplica Integración Continua a todos nuestros proyectos internos. Cuando trabajamos con clientes, damos prioridad a los deseos y requisitos del cliente con respecto al proceso de desarrollo de software. Si su propio proceso permite el uso inicial de recursos adicionales para configurar la integración continua correctamente, le recomendamos que la aplique a su proyecto de desarrollo de software con nosotros. Con la Integración Continua exitosa en su lugar, podemos trabajar juntos para el siguiente paso, permitiendo que sus lanzamientos sean cada vez más flexibles con la Entrega Continua.
